Devices that have an _HID (hardware identification) object are generally enumerated by and have their drivers loaded by ACPI.  Devices with an _ADR (physical address) object are not generally enumerated by nor have their drivers loaded by ACPI, but when necessary ACPI can evaluate such objects to perform required functions.  Devices that are enumerable on other buses, like PCI or USB devices, have an encoding technique that allows ACPI to locate their bus-specific addresses so they can be referenced in ACPI.  ACPI uses two types of data structures to share firmware information with the OS: data tables and definition blocks. Data tables contain raw data from hardware specifications that get used directly by device drivers. Definition blocks are created by writing ACPI Source Language (ASL) code and converting that to ACPI Machine Language (AML) bytecode, which can then be processed by an AML interpreter.   The AML interpreter in the OS extracts AML bytecode from definition blocks and converts that into enumerable objects, which together form an OS construct called the ACPI Namespace. Objects in the Namespace may have directly defined values, or can be used to perform loop constructs, evaluate conditions, and perform other operations that applications require. The compiled Namespace is a hierarchical representation of the ACPI devices in a system, and the system bus is the root of enumeration for these devices.   The first table used by the ACPI subsystem is the eXtended Root System Description table (XSDT), which points to the Fixed ACPI Description Table (FADT), the Secondary System Description Tables (SSDTs), and other ACPI tables. The FADT directs the ACPI subsystem to the Differentiated System Description Table (DSDT), which is essential for ACPI because it is the first table containing a definiton block. The FADT also points to the Firmware ACPI Control Structure (FACS).   ACPI has a runtime model to handle any ACPI interrupt events that occur during system operation. ACPI continues to evaluate Namespace objects as needed to handle interrupt events.  ACPI includes a thermal event model to allow systems to control their temperature either actively (for example by turning on a fan) or passively (for example by limiting the amount of power used by the processor).
